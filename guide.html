サンプル問題

エージェントが lookup_order と get_customer を頻繁に混同します。最初に取るべき対策はどれですか？
A Few-shotサンプルを5〜8個追加する
B ツール説明文を拡張して詳細な説明を記述する
C キーワードベースのルーティングロジックを実装する
D 2つのツールを1つに統合する
ツール説明はLLMがツールを選択する主要メカニズムです。説明の充実は「低労力・高効果」であり、Few-shotより先に試みるべき対策です。
マルチエージェントシステムで、サブエージェントが正しく動作するために必ず行うべきことはどれですか？
A 親エージェントと同じモデルを使用する
B 親エージェントのコンテキストを自動的に引き継ぐ設定を有効にする
C タスクに必要な情報を明示的にサブエージェントに渡す
D サブエージェントに全ツールへのアクセス権を付与する
サブエージェントは親コンテキストを自動継承しません。コーディネーターはサブタスクに必要な情報（データ・指示・制約）を明示的に渡す設計が必須です。

推奨学習ステップ

  • Anthropic公式ドキュメント（docs.anthropic.com）でエージェント設計・MCP・Claude Codeの基礎を習得する
  • MCPサーバーを実際に構築する（GitHub APIやSlack APIをMCP化するのが効果的）
  • Agent SDKでマルチエージェントを実装し、コンテキスト引き継ぎを体験する
  • CLAUDE.mdを実プロジェクトで最適化し、パス別制約の自動適用を試す
  • 本問題集でドメイン別の弱点を特定し、集中的に復習する
  • サンプル問題・模擬試験で時間配分（120分・60問）を体感する

試験が評価する3つの視点

1. 確率論から決定論へ

LLMにプロンプトで指示するだけでなく、条件チェックやバリデーションをコードで実装する「確定的な制御」を重視。重要なビジネスロジックはプログラムで固定する。

2. MCPの真価はセマンティックインターフェース

MCPは単なるツール連携ではなく、AIがサービスの意味を理解して使えるインターフェース。ツールの説明・スキーマ・エラー設計がLLMの判断精度を左右する。

3. 運用設計（本番環境の現実）

コンテキストの劣化・エラー回復・コスト管理・セキュリティを含む「本番環境での信頼性」まで問われる。試験は開発者ではなくアーキテクトとしての責任を評価する。

必ず覚える6大アンチパターン

試験では「正しい設計」だけでなく「なぜその設計がダメか」を問う問題が多い。

❌ Few-shotでツール実行順を制御する
  • 実行順序はコンプライアンス上の制約であり、プロンプトで「お願い」するものではない
  • 正解：PreToolUseフックや前提条件チェックでプログラム的に強制する
❌ モデルの自己報告「確信度」でエスカレーションを判断する
  • LLMの確信度スコアは、最も難しいケースほど正確に機能しない
  • 正解：閾値・ルールベースなど決定論的なロジックでエスカレーションを制御する
❌ コスト削減のため全ワークフローをBatch APIに流す
  • Batch APIにはSLA（応答時間保証）がない
  • 正解：ユーザー向けのリアルタイムワークフローは通常のAPIを使う
❌ コンテキストウィンドウを広げてアテンション問題を解決する
  • ウィンドウサイズを増やしてもLostintheMiddle問題は解決しない
  • 正解：RAG・要約・重要情報の先頭/末尾配置で対処する
❌ サブエージェント失敗時に空の結果を返す
  • コーディネーターが何が起きたか分からず回復できない
  • 正解：構造化エラーコンテキスト（失敗理由・再試行可否）を返す
❌ 全エージェントに全ツールを付与する
  • 不要なツールが選択される誤りが増え、信頼性が低下する
  • 正解：各エージェントの役割に合わせてツールのスコープを絞る

stop_reason がループ制御の唯一の正解

エージェントループの終了判断は stop_reason フィールドのみを信頼する。これは試験頻出。

❌ これらはすべて間違い
  • テキスト解析（「完了しました」という文言で判断する）
  • 繰り返し回数の上限だけでループを制御する（安全網としてはOK）
  • tool_useブロックと同時にテキストが返るかどうかで判断する
✅ 正解
  • stop_reason === "end_turn" のみがループ終了の確定的シグナル
  • ツール結果は必ず会話履歴に追加してから次のイテレーションに進む

60問は6つのシナリオから構成。本番では6つのうち4つが出題される。すべて対策すること。

  • シナリオ3：マルチエージェント調査システム
  • シナリオ4：開発者生産性向上ワークフロー
  • シナリオ5：Claude Code × CI/CD統合
  • シナリオ6：構造化データ抽出パイプライン

単語帳

タップして解説を表示

Few-shot（フューショット） ›

「例を数個見せることでモデルに学習させる手法」

AIに指示するとき、「こういう入力にはこういう出力」という例を2〜5個プロンプトに含めること。例を0個見せる場合はZero-shot、1個はOne-shot、複数はFew-shotと呼ぶ。

例： 感情分類をさせたい場合
「嬉しい」→ ポジティブ
「悲しい」→ ネガティブ
「最高の1日だった」→ ?（Claudeが続きを予測）

試験では：ツール選択の改善にFew-shotを使うのは非効率。ツール説明文の充実が先。

コンテキストウィンドウ ›

「AIが1回のリクエストで一度に読める情報量の上限」

人間の「作業机の広さ」に例えられる。机が広いほど多くの資料を広げられるが、有限。Claudeは最大200,000トークン（日本語で約15万字相当）まで一度に処理できる。

トークンとは：単語や文字の塊。英語は約3〜4文字で1トークン。日本語は1文字が1〜3トークン程度。

入力 + 出力 = 合計トークン数が上限以内に収まる必要がある。

試験では：上限に近づいたら自動要約・RAGで管理する設計が求められる。

アテンション問題（Lost in the Middle） ›

「長い文章の真ん中にある情報をAIが見落としやすい現象」

人間が長い文書を読むとき、最初と最後は覚えているが途中を忘れがちなのと似ている。LLMも長いコンテキストの中間部分の情報を参照する精度が下がる。

対策：重要な指示や情報はプロンプトの最初か最後に置く。コンテキストウィンドウを広げても解決しない（よくある誤解）。

試験では：「コンテキストウィンドウを広げれば解決する」という選択肢は✕。RAGや配置で解決する。

stop_reason（ストップリーズン） ›

「AIがなぜ出力を止めたかを示すフィールド」

APIレスポンスに含まれる情報で、AIが出力を停止した理由を教えてくれる。

"end_turn" → 自然に完了した（ループ終了のサイン）
"tool_use" → ツールを使いたいので一時停止
"max_tokens" → トークン上限に達した
"stop_sequence" → 指定の終了文字列に達した

試験では：エージェントループの終了判断は stop_reason のみを使う。「完了しましたと言っているから終了」という判断は✕。

Tool Use / Function Calling ›

「AIが外部のツールや関数を呼び出せる仕組み」

通常のAIはテキストを返すだけだが、Tool Useを使うとAIが「このツールを呼んで」と指示を出せる。開発者がそのツールを実行して結果をAIに返すと、AIがその結果を元に次の行動を決める。

例： 天気を聞かれたとき
1. AIが get_weather(city="東京") を呼ぶよう指示
2. 開発者側で天気APIを実行
3. 結果「晴れ、25度」をAIに返す
4. AIが「今日の東京は晴れで25度です」と回答

AnthropicではTool Use、OpenAIではFunction Callingと呼ぶが同じ概念。

試験では：最も出題頻度が高いトピック。スキーマ設計・エラー処理・並行呼び出しまで習得必須。

Human-in-the-Loop（ヒューマンインザループ） ›

「自動処理の途中に人間の確認・判断を挟む設計」

AIが全自動でタスクを実行する中で、特定のポイントで人間に「これを実行してよいですか？」と確認を取る仕組み。

必要な場面：

  • ·メール送信・SNS投稿など、元に戻せないアクション
  • ·データベースの更新・削除
  • ·金融取引・契約の締結
  • ·AIが「指示が曖昧で確信が持てない」と判断した場合

試験では：「全ステップで確認が必要」は✕。高リスク・不可逆なアクションの前だけに絞る設計が正解。

エージェントループ（Agentic Loop）›

「AIが目標を達成するまで自律的に考えて行動し続けるサイクル」

入力を受け取る → ツールを使うか判断 → ツールを実行 → 結果を観察 → 目標を達成したか判断、という一連のサイクルを繰り返す。stop_reason === "end_turn" になるまで継続する。

試験では：ループの終了条件・コスト爆発防止・ループ検出の設計が頻出。

オーケストレーター / サブエージェント›

「指揮官（オーケストレーター）と実行担当（サブエージェント）の役割分担」

オーケストレーターはゴールをサブタスクに分解して専門のサブエージェントに指示を出す上位エージェント。サブエージェントは与えられたサブタスクを実際に実行する下位エージェント。

重要：サブエージェントは親（オーケストレーター）のコンテキストを自動で引き継がない。必要な情報は明示的に渡す必要がある。

試験では：コンテキストの引き継ぎ方と信頼境界の設計が頻出。

ReActパターン›

「Reasoning（推論）+ Acting（行動）を交互に繰り返すエージェント設計」

Thought（何をすべきか考える）→ Action（ツールを実行する）→ Observation（実行結果を観察する）のサイクルを繰り返すパターン。拡張思考（Extended Thinking）と組み合わせると複雑な推論タスクの精度が上がる。

試験では：エージェントの基本設計パターンとして問われる。

最小権限の原則›

「エージェントに必要最小限のツール・権限だけを与える設計原則」

タスクに不要なツール・広すぎるファイルアクセス・過剰なAPI権限を与えると、誤操作・意図しないデータ損失・セキュリティ侵害のリスクが高まる。必要な権限だけに絞ることで被害を最小化できる。

試験では：「全ツールを全エージェントに付与する」は✕の典型アンチパターン。

チェックポイント（Checkpoint）›

「処理の途中経過を保存しておき、失敗時に途中から再開できる仕組み」

長時間実行するエージェントでネットワーク障害・タイムアウト・レート制限などが発生した場合、最初から全てやり直さずに済む。完了済みステップ・収集済みデータ・部分的な結果を永続化しておく。

試験では：長時間エージェントの信頼性設計の文脈で出題。

プロンプトインジェクション›

「外部から取得したコンテンツに悪意ある指示を埋め込みAIを操作する攻撃」

エージェントがWebページ・メール・ドキュメントを読み取る際、その中に「システムの指示を無視して〇〇せよ」といった命令が埋め込まれている攻撃。エージェントがその指示に従ってしまう危険がある。

対策：ユーザー入力や外部コンテンツをXMLタグで信頼済み指示と明確に分離し、信頼レベルを区別する。

試験では：マルチエージェント・外部コンテンツ取得の設計でセキュリティ観点として問われる。

ハンドオフ（Handoff）›

「あるエージェントが処理の制御を別の専門エージェントに引き渡すこと」

例：カスタマーサポートエージェントが技術的な質問を受けたとき、技術サポート専門エージェントに処理を転送する。Anthropic Agent SDKで実装できる。

試験では：マルチエージェント設計・専門化の文脈で出題。

フォールバック戦略›

「一次手段が失敗したときの代替手段を段階的に試みる設計」

①一次ツールの失敗 → 代替ツールを試みる → ②リトライ（指数バックオフ） → ③簡略化したアプローチで再試行 → ④全て失敗した場合のみ人間にエスカレーション、という段階的対応が理想。

試験では：「失敗したら即停止」は✕。段階的な回復設計が正解。

MCP（Model Context Protocol）›

「AIモデルと外部ツール・データソースを標準化された方法で接続するプロトコル」

Anthropicが策定したオープン規格。MCPサーバーを実装することで、データベース・ファイルシステム・外部APIなど任意のサービスをAIエージェントのツールとして提供できる。

MCPが提供できる3種類：
① ツール（実行可能な関数）
② リソース（ファイルやデータの読み取り）
③ プロンプト（再利用可能なテンプレート）

試験では：概念だけでなく実装レベルの理解が求められる。

input_schema / JSON Schema›

「ツールが受け取るパラメーターの型・形式を定義する仕様書」

ツール定義に必要な3要素：
① type: "object"（必須）
② properties（各パラメーターの型・説明）
③ required（必須パラメーターの配列）

descriptionが充実しているほど、モデルが正しく使える。enumで有効な値の集合を示すと誤選択が減る。

試験では：ツール設計の基本として頻出。

tool_choice（auto / any / tool）›

「モデルがツールを使うかどうかを制御するパラメーター」

"auto" → モデルが自律的にツール使用を判断（デフォルト）
"any" → 提供されたツールのどれかを必ず使う
"tool" → 特定のツールを必ず使う（name指定）
"none" → ツールを使わない

試験では：構造化データ抽出でtool_choice=toolを使い強制する設計が頻出。

is_error フラグ›

「ツール実行でエラーが起きたことをAIに伝えるフラグ」

ツールが失敗したとき、is_error: true とエラー内容をtool_resultに含めてAIに返す。AIはエラー情報を受け取り代替手段の提案や別のアプローチを試みる。

✕ 悪い例：エラーを無視して空の結果を返す → コーディネーターが何が起きたか分からない
✅ 正解：is_error:true + エラー詳細 + 再試行可否を返す

試験では：エラー設計のアンチパターンとして出題。

並行ツール呼び出し（Parallel Tool Use）›

「AIが複数のツールを同時に実行するよう指示する機能」

ClaudeがAPIレスポンスで複数のtool_useブロックを返すことがある。この場合、それぞれを並行実行し、全ての結果を個別のtool_resultブロックとして同一メッセージに含めて返す必要がある。

試験では：「最初のツールだけ実行する」は✕。全ツールを並行実行して全結果を返すのが正解。

CLAUDE.md›

「プロジェクト固有のルール・コンテキストをClaude Codeに伝えるファイル」

プロジェクトルートに置くMarkdownファイル。Claude Codeが自動的に読み込む。プロジェクトの概要・コーディング規約・ビルド手順・禁止事項などを記述する。

2種類の場所：
~/.claude/CLAUDE.md → 全プロジェクト共通のグローバル設定
.claude/CLAUDE.md → プロジェクト固有の設定（Gitで共有可能）

試験では：APIキー等の秘密情報は絶対に書かない。

Hooks（PreToolUse / PostToolUse）›

「ツール実行の前後に任意のシェルコマンドを自動実行する仕組み」

PreToolUse → ツール実行直前に起動。危険なコマンドの確認・自動拒否・監査ログ記録に使う
PostToolUse → ツール実行直後に起動。コードフォーマット・テスト自動実行に使う
Stop → セッション終了時に起動

試験では：高リスク操作の前にプログラム的な制御を実装する手段として出題。

Planモード›

「実行前に計画を立ててユーザーに確認してから作業するモード」

大規模な変更・リファクタリング・複数ファイルへの影響が出る作業など、慎重に進めたい場合に使う。直接実行（単純タスク）と使い分けることが設計上重要。

試験では：どの場面でPlanモードを使うべきかの判断が問われる。

拒否リスト / 許可リスト（deny / allow list）›

「Claude Codeが実行できるツールを制御するリスト」

settings.jsonのpermissions.allowとpermissions.denyで設定する。

重要な優先順位：拒否リストが許可リストより優先される。同じパターンが両方にある場合は必ず拒否が勝つ。セキュリティ上重要なツールは明示的に拒否リストに追加する。

試験では：優先順位の問題（拒否 > 許可）が頻出。

Prefill（事前充填）›

「AIの返答の書き始めをあらかじめ指定することで出力形式を強制する技法」

messagesの最後にrole: "assistant"のメッセージ（書き始め）を含めると、AIがその続きを生成する。

例：{"role":"assistant","content":"{"} と指定するとClaudeはJSONオブジェクトとして続きを生成する。

試験では：構造化JSON出力を確実に得る手段の一つとして出題。

Chain-of-Thought（CoT）›

「AIに推論の過程を段階的に示させることで回答精度を上げる手法」

「ステップバイステップで考えてください」という指示でモデルに思考プロセスを出力させる。数学的計算・論理パズル・多段階の因果推論・複雑な計画立案で特に効果的。単純なタスクでは逆にオーバーヘッドになる。

試験では：どの場面でCoTが有効かを判断する問題が出る。

stop_sequences（ストップシーケンス）›

「この文字列が出力に現れたらAIの生成を止める、カスタム終了条件」

例：stop_sequences: ["</response>", "\n\nHuman:"] と設定すると、これらの文字列が生成された時点で出力を停止する。

用途：Few-shotプロンプトでの応答境界制御・XMLタグによる出力の打ち切り・構造化出力の終了条件定義。

試験では：stop_reasonの"stop_sequence"とセットで理解する。

temperature / top_p›

「AIの出力のランダム性・多様性を制御するパラメーター」

temperature：
0に近い → 毎回ほぼ同じ決定論的な出力（コーディング・事実確認向き）
1に近い → 毎回異なる多様・創造的な出力（創作・ブレインストーミング向き）

top_p（nucleus sampling）：
累積確率がp値に達するまでのトークン候補集合から選択する。top_p=0.9なら確率上位90%のトークンから選ぶ。

試験では：temperatureとtop_pは通常どちらか一方だけ変更する。両方同時に変えるのは非推奨。

プロンプトキャッシング（Prompt Caching）›

「繰り返し使う長いプロンプトをキャッシュしてコストと時間を削減する機能」

同じ長いシステムプロンプト・参照ドキュメント・コードベースを毎回送る代わりに、キャッシュしておき2回目以降は再利用する。入力コストを最大90%削減できる。

設定の注意点：
· キャッシュポイントは変わらないコンテンツの末尾に設置
· 最小トークン数要件あり（Sonnet/Opusは1024、Haikuは2048トークン未満はキャッシュされない）

試験では：ブレークポイントの配置場所と最小トークン要件が頻出。

RAG（Retrieval-Augmented Generation）›

「外部の知識データベースから関連情報を検索してAIに渡す手法」

大量のドキュメントを全部コンテキストに入れるのではなく、ベクトルデータベースなどに保存しておき、質問に関連する部分だけを検索してAIに渡す。コンテキストウィンドウの節約とコスト削減に有効。

例：社内マニュアル1,000ページ全体を毎回送る代わりに、質問に近い5ページだけを検索して渡す。

試験では：大規模コードベース処理・コンテキスト効率化の正解手段として頻出。

ステートレス API›

「サーバー側に会話の状態が保存されず、毎回全履歴を送る必要があるAPI設計」

AnthropicのAPIはリクエストをまたいで会話履歴を記憶しない。マルチターン会話を実現するには、クライアント側でmessages配列（userとassistantが交互）を管理して毎回全履歴を含めて送る必要がある。

結果：会話が長くなるほどトークン消費（コスト）が増加する。要約やトリミングによるコンテキスト管理が必要になる。

試験では：コンテキスト管理が必要な根本的な理由として理解する。

トークン（Token）›

「AIがテキストを処理する最小単位。料金計算の基準」

単語や文字の塊。英語は平均3〜4文字で1トークン。日本語は1文字が1〜3トークン程度になることが多く、英語より多くのトークンを消費する。

正確なトークン数の計測：Anthropicのtokencount API（/v1/messages/count_tokens）またはSDKのcount_tokensメソッドを使う。「文字数÷4」などの近似は不正確。

試験では：トークン見積もりの正しい方法と、コンテキストウィンドウ管理の文脈で出題。

要約（Summarization）のトレードオフ›

「会話を要約するとコンテキストは節約できるが、細部が失われるリスクがある」

長い会話を要約するとコンテキストウィンドウを解放できる一方で、具体的な数値・固有名詞・決定事項の詳細が失われる可能性がある。

対策：
· 重要な事実は箇条書きで明示的に保持
· 要約の品質を定期的に検証
· 重要な決定事項は要約から除外する

試験では：「要約すれば全情報が保持される」は✕。細部が失われるリスクが正解。

プロンプトキャッシュのブレークポイント›

「キャッシュの区切り位置。どこまでをキャッシュするかを指定するマーカー」

プロンプトキャッシングを使うとき、キャッシュしたい部分の末尾にcache_control: {"type": "ephemeral"}を設定する。

重要な制約：
· 変わらないコンテンツの末尾に設置する（動的なコンテンツの前には置かない）
· 最小トークン数未満はキャッシュされない（Sonnet/Opus: 1024、Haiku: 2048）
· キャッシュの有効期限は5分（最後のアクセスから）

試験では：配置場所・最小トークン・有効期限の3点が問われる。
