Anthropicが提供する初の公式技術認定資格。単なるプロンプトエンジニアリングではなく、Claudeを使って現実の複雑な課題を解決するための実践的設計能力を評価します。ソリューションアーキテクト・AI実装エンジニアが対象です。

ドメイン別 重要ポイント

Agentic Architecture 27%
  • ハブ&スポーク型マルチエージェント設計（コーディネーター＋専門サブエージェント）
  • ReActパターン：Thought → Action → Observation のサイクル
  • サブエージェントは親コンテキストを自動継承しない — 必要情報は明示的に渡す
  • 最小権限の原則：タスクに必要最小限のツール・権限のみ付与
  • Human-in-the-Loop：不可逆・高リスクなアクション前に人間確認
  • コスト爆発防止：最大ステップ数・タイムアウト・ループ検出を実装
  • プロンプトインジェクション対策：外部コンテンツからの指示を隔離
Tool Design & MCP 18%
  • ツール説明文がLLMの選択判断を左右する最重要要素（Few-shotより効果的）
  • input_schemaはJSON Schema形式：type・properties・requiredを必ず定義
  • MCP設定：.mcp.json（プロジェクト共有）と~/.claude.json（個人用）
  • エラー設計：is_error:true フラグと詳細メッセージでClaudeに通知
  • 並行ツール呼び出し：全結果を個別のtool_resultブロックで同一メッセージに返す
  • ツールの粒度：細かすぎず粗すぎず、ユースケースに適した設計
  • 長い結果はトリミング・要約でコンテキストウィンドウを節約
Claude Code & Workflows 20%
  • CLAUDE.md：プロジェクトルールをClaude Codeに自動読み込みさせる設定ファイル
  • .claude/rules/：パス別制約を自動適用
  • Planモード（大規模変更）vs 直接実行（単純タスク）の使い分け
  • Hooksで PreToolUse / PostToolUse / Stop イベントにシェルコマンドを紐付け
  • 拒否リストが許可リストより優先される（セキュリティ上重要）
  • claude -p（printモード）でCI/CDパイプラインに組み込み
  • カスタムスラッシュコマンド：.claude/commands/にMarkdownファイルを配置
Prompt Engineering 20%
  • XMLタグによる構造化：Claudeはタグの区切りを明確に認識
  • Prefill技法：assistantロールの書き始めを提供して出力形式を強制
  • Chain-of-Thought：多段階推論・計算・論理パズルで精度向上
  • JSONスキーマ活用 + tool_choice=required で構造化出力を確実に取得
  • 否定指示より肯定指示：「何をしてほしいか」を明確に定義
  • Lost in the Middle対策：重要情報はプロンプトの最初か最後に配置
  • プロンプトキャッシング：繰り返し使う長いコンテンツでコスト最大90%削減
Context Management 15%
  • APIはステートレス：各リクエストに全会話履歴を含める必要がある
  • コンテキスト上限70〜80%到達時に自動要約・圧縮を実装
  • RAG：大規模ドキュメントから関連部分のみを動的に取得
  • 要約時は重要な数値・固有名詞・決定事項を明示的に保持
  • プロンプトキャッシュのブレークポイント：最小トークン数要件を確認
  • ストリーミングはコンテキスト使用量に影響しない

サンプル問題

エージェントが lookup_order と get_customer を頻繁に混同します。最初に取るべき対策はどれですか？
A Few-shotサンプルを5〜8個追加する
B ツール説明文を拡張して詳細な説明を記述する
C キーワードベースのルーティングロジックを実装する
D 2つのツールを1つに統合する
ツール説明はLLMがツールを選択する主要メカニズムです。説明の充実は「低労力・高効果」であり、Few-shotより先に試みるべき対策です。
マルチエージェントシステムで、サブエージェントが正しく動作するために必ず行うべきことはどれですか？
A 親エージェントと同じモデルを使用する
B 親エージェントのコンテキストを自動的に引き継ぐ設定を有効にする
C タスクに必要な情報を明示的にサブエージェントに渡す
D サブエージェントに全ツールへのアクセス権を付与する
サブエージェントは親コンテキストを自動継承しません。コーディネーターはサブタスクに必要な情報（データ・指示・制約）を明示的に渡す設計が必須です。

推奨学習ステップ

  • Anthropic公式ドキュメント（docs.anthropic.com）でエージェント設計・MCP・Claude Codeの基礎を習得する
  • MCPサーバーを実際に構築する（GitHub APIやSlack APIをMCP化するのが効果的）
  • Agent SDKでマルチエージェントを実装し、コンテキスト引き継ぎを体験する
  • CLAUDE.mdを実プロジェクトで最適化し、パス別制約の自動適用を試す
  • 本問題集でドメイン別の弱点を特定し、集中的に復習する
  • サンプル問題・模擬試験で時間配分（120分・60問）を体感する

試験が評価する3つの視点

1. 確率論から決定論へ

LLMにプロンプトで指示するだけでなく、条件チェックやバリデーションをコードで実装する「確定的な制御」を重視。重要なビジネスロジックはプログラムで固定する。

2. MCPの真価はセマンティックインターフェース

MCPは単なるツール連携ではなく、AIがサービスの意味を理解して使えるインターフェース。ツールの説明・スキーマ・エラー設計がLLMの判断精度を左右する。

3. 運用設計（本番環境の現実）

コンテキストの劣化・エラー回復・コスト管理・セキュリティを含む「本番環境での信頼性」まで問われる。試験は開発者ではなくアーキテクトとしての責任を評価する。

必ず覚える6大アンチパターン

試験では「正しい設計」だけでなく「なぜその設計がダメか」を問う問題が多い。

❌ Few-shotでツール実行順を制御する
  • 実行順序はコンプライアンス上の制約であり、プロンプトで「お願い」するものではない
  • 正解：PreToolUseフックや前提条件チェックでプログラム的に強制する
❌ モデルの自己報告「確信度」でエスカレーションを判断する
  • LLMの確信度スコアは、最も難しいケースほど正確に機能しない
  • 正解：閾値・ルールベースなど決定論的なロジックでエスカレーションを制御する
❌ コスト削減のため全ワークフローをBatch APIに流す
  • Batch APIにはSLA（応答時間保証）がない
  • 正解：ユーザー向けのリアルタイムワークフローは通常のAPIを使う
❌ コンテキストウィンドウを広げてアテンション問題を解決する
  • ウィンドウサイズを増やしてもLostintheMiddle問題は解決しない
  • 正解：RAG・要約・重要情報の先頭/末尾配置で対処する
❌ サブエージェント失敗時に空の結果を返す
  • コーディネーターが何が起きたか分からず回復できない
  • 正解：構造化エラーコンテキスト（失敗理由・再試行可否）を返す
❌ 全エージェントに全ツールを付与する
  • 不要なツールが選択される誤りが増え、信頼性が低下する
  • 正解：各エージェントの役割に合わせてツールのスコープを絞る

stop_reason がループ制御の唯一の正解

エージェントループの終了判断は stop_reason フィールドのみを信頼する。これは試験頻出。

❌ これらはすべて間違い
  • テキスト解析（「完了しました」という文言で判断する）
  • 繰り返し回数の上限だけでループを制御する（安全網としてはOK）
  • tool_useブロックと同時にテキストが返るかどうかで判断する
✅ 正解
  • stop_reason === "end_turn" のみがループ終了の確定的シグナル
  • ツール結果は必ず会話履歴に追加してから次のイテレーションに進む

本番で出題される6つのシナリオ

60問は6つのシナリオから構成。本番では6つのうち4つが出題される。すべて対策すること。

  • シナリオ1：カスタマーサポート解決エージェント
  • シナリオ2：Claude Codeによるコード生成
  • シナリオ3：マルチエージェント調査システム
  • シナリオ4：開発者生産性向上ワークフロー
  • シナリオ5：Claude Code × CI/CD統合
  • シナリオ6：構造化データ抽出パイプライン

985点取得者のアドバイス

  • Tool Use / Function Callingが最も出題される — 徹底的に習得する
  • MCPは「概念として知っている」ではなく実装レベルで理解する
  • コンテキスト最適化は理論ではなく実践的に問われる（いつ・何を削るか）
  • Human-in-the-Loopは「いつ・どのように」挿入するかをシナリオで問われる
  • 試験は完全クローズドブック — AIアシスタント・ネット検索は一切不可
  • 経験者は20〜30時間（2〜4週間）の集中学習で合格できる

出題されない範囲（スコープ外）

モデルの内部構造・ファインチューニング・料金体系・競合製品・一般的なML理論・インフラ設定・UI設計・詳細なコンプライアンス規制
